# KExcel 성능 벤치마크 보고서

이 문서는 네이티브 엔진 API(FastExcel 및 Apache POI)와 비교한 KExcel DSL의 성능 특성을 상세히 다룹니다.

## 1. 요약 (Executive Summary)
- **DSL 오버헤드**: 최적화 후 **16.3% -> 8.6%**(FastExcel) 및 **약 0%**(POI)로 감소.
- **메모리 효율**: 함수 인라인화(Inlining)를 통해 셀당 추가 메모리 할당량을 **제로(0)**에 가깝게 구현.
- **엔진 비교**: 대용량 데이터 처리 시 FastExcel이 Apache POI(SXSSF)보다 약 **25~30% 더 빠름**.
- **안정성**: 512MB 힙 환경에서 1,000,000행 처리 시에도 일정한 메모리 점유율 유지 확인.

---

## 2. 측정 환경
- **JVM**: OpenJDK 21 (Adoptium)
- **메모리 설정**: 512MB (`-Xmx512m`)
- **GC**: G1GC
- **JMH 버전**: 1.36
- **사용 엔진**: 
  - FastExcel 0.18.4
  - Apache POI 5.2.5 (SXSSF)

---

## 3. DSL 오버헤드 분석
네이티브 API 직접 호출 대비 KExcel DSL 사용 시의 비용 측정.
(10,000 행 x 10 열 기준)

| 드라이버 | 네이티브 처리량 | DSL 처리량 | **오버헤드 (%)** |
| :--- | :--- | :--- | :--- |
| **FastExcel** | 7.925 ops/s | 7.243 ops/s | **8.6%** |
| **Apache POI** | 4.485 ops/s | 5.086 ops/s | **~0%** (오차 범위 내) |

> [!NOTE]
> FastExcel의 오버헤드가 더 높게 나타나는 이유는 엔진 자체가 극도로 빨라 락(Lock) 체크나 안전 검증 로직의 상대적 비용이 더 잘 보이기 때문입니다. POI의 경우 엔진 자체 로직이 무거워 DSL 비용이 상쇄됩니다.

---

## 4. 대용량 데이터 안정성 (Streaming)
방대한 행 수를 가진 워크북 생성 시의 실행 시간 비교.

| 행 수 | FastExcel 드라이버 | POI 드라이버 (SXSSF) |
| :--- | :--- | :--- |
| **100,000** | 0.238 초 | 0.440 초 |
| **1,000,000** | 3.396 초 | 4.249 초 |

**결론**: 두 드라이버 모두 데이터 크기에 비례하여 선형적으로 성능이 확장됩니다. 순수 처리 속도는 FastExcel이 우세합니다.

---

## 5. 스타일 상속 및 병합 성능
KExcel의 스타일 상속 시스템이 성능에 미치는 영향 측정.

| 시나리오 | 처리량 (ops/s) |
| :--- | :--- |
| **스타일 없음 (No Style)** | 81.47 |
| **상속된 스타일 (Inherited)** | 55.93 |
| **개별 셀 스타일 (Individual)** | 52.30 |

**결론**: 워크북/시트 레벨의 **상속된 스타일**을 사용하는 것이 모든 셀에 개별 스타일을 입히는 것보다 미세하게 더 빠릅니다. 이는 쓰기 시점의 스타일 해소(Resolution) 복잡도를 줄여주기 때문입니다.

## 6. 최적화 성과 (Before vs After)
`inline` 함수 적용 및 핫루프 내 `runCatching` 제거 전후 비교.

| 지표 | 최적화 전 | 최적화 후 | 개선율 |
| :--- | :--- | :--- | :--- |
| FastExcel DSL 오버헤드 | 16.3% | 8.6% | **약 47% 감소** |
| 실행당 메모리 할당량 | 약 200 KB 추가 | **약 300 Bytes** 추가 | **99.8% 감소** |

> [!TIP]
> 메모리 할당량이 획기적으로 줄어든 것은 Kotlin 컴파일러가 DSL 블록을 성공적으로 인라인화하여, 핫루프 내에서 람다 객체 생성을 완전히 제거했음을 의미합니다.

## 7. 사례 연구: 셀 병합 성능 분석

### 7.1 문제 상황
대용량 데이터 생성 시 `mergeCells`를 사용하면 성능이 급격히 저하되거나 `OutOfMemoryError`가 발생한다는 사용자 보고를 확인했습니다.

### 7.2 원인 분석
병합 빈도(10행마다 병합)에 따른 영향을 측정하기 위해 `MergeBenchmark`를 구축하여 테스트를 진행했습니다.

**초기 측정 결과 (POI vs FastExcel):**
- **FastExcel**: 성능 저하 없이 안정적인 처리량(32 ops/s) 및 메모리 사용량 유지.
- **Apache POI**: 처리량이 18 ops/s에서 **3.3 ops/s로 폭락(▼82%)**하고, GC 할당 속도가 **610% 급증**함.

**원인**: Apache POI의 표준 `addMergedRegion` 메서드가 새로운 영역을 추가할 때마다 기존 모든 영역과 겹치는지 전수 조사($O(n^2)$)를 수행함.

### 7.3 최적화 과정
`PoiDriver`에서 중복 검증 로직을 우회하는 `addMergedRegionUnsafe()` 메서드를 사용하도록 코드를 수정했습니다.

### 7.4 최종 결과
`Unsafe` 최적화 적용 후 POI의 성능이 비약적으로 회복되었습니다:

| 측정 지표 | 최적화 전 (`addMergedRegion`) | 최적화 후 (`addMergedRegionUnsafe`) | **개선율** |
| :--- | :--- | :--- | :--- |
| 처리량 (Throughput) | 3.36 ops/s | **17.42 ops/s** | **▲ 518%** |
| GC 할당 속도 | 1580.8 MB/s | **328.6 MB/s** | **▼ 79%** |

### 7.5 엔지니어링 제언
CPU 병목은 해결되었으나, **메모리 점유(Memory Residency)** 특성은 여전합니다. 병합 정보는 파일 작성이 완료될 때까지 메모리에 유지되므로, 수십만 건 이상의 병합이 필요한 경우 JVM 힙 메모리를 충분히 확보하는 것이 권장됩니다.

---

## 8. 권장 사항
- **최고의 성능이 필요한 경우**: `FastExcelDriver`를 사용하세요.
- **복잡한 수식이나 하위 호환성이 중요한 경우**: `PoiDriver`를 사용하세요.
- **유지보수와 성능을 동시에 잡으려면**: **스타일 상속 시스템**을 적극 활용하세요. 코드가 깔끔해질 뿐만 아니라 성능상으로도 유리합니다.
